## Change details

### Fix 1 — Removed dead hook (CreateShipmentService.php)

Removed line:
```php
do_action('dexpress_shipment_created', new ShipmentCreated($shipment, $shipment->orderId));
```
Retained only:
```php
do_action('dexpress/shipment.created', new ShipmentCreated($shipment, $shipment->orderId));
```
`EmailNotificationSubscriber` and `SimulationService` both listen on `dexpress/shipment.created`. No code listens on the old underscore hook.

### Fix 2 — Removed `$_bucket` parameter (ShipmentStatusOrderNoteFormatter.php + ShipmentStatusIngestionService.php)

Signature changed from:
```php
public static function format(string $packageCode, string $labelSnapshot, StatusEmailBucket $_bucket): string
```
To:
```php
public static function format(string $packageCode, string $labelSnapshot): string
```
Call site in `ShipmentStatusIngestionService::applyInboundStatus()` updated to drop the third argument. The unused `use` statement for `StatusEmailBucket` was also removed from the formatter file.

### Fix 3 — Removed `label()` from StatusEmailBucket enum

The method returned translated strings for each case but was never invoked. The admin orders column uses `badgeModifierClass()` + CSS for display. Status labels shown to users come from the `wp_dexpress_status_codes` table via `StatusCodeRepository::resolveOfficialShipmentStatusLabel()`.

### Fix 4 — Removed PHASE 2B milestone comments (CreateShipmentService.php)

Removed the 4-line block comment at the top of the file. All described work (Package Shop dispenser flow, address validation, checkout server-side validation) is implemented and shipped.

### Fix 5 — Removed PHASE 2B checklist comment (package-shop-checkout.js)

Removed the 5-line `// PHASE 2B HOOKS — ready for implementation` comment block. Dispenser selection, hidden field population, and pre-submit validation are all implemented in the file.

### Fix 6 — Documented EARLY_PROBLEM_SIDS (ShipmentEmailRenderContextFactory.php)

Added a block docblock explaining:
- **Purpose:** sIDs where the problem occurred before courier collected the package; controls which progress step (1 = pickup vs 2 = transit) shows the problem marker in shipment notification emails
- **Grouping:** comments on each line cluster explaining what each group represents at a business level
- **Authority:** official label for each sID is resolved at runtime from `wp_dexpress_status_codes`

## Overall codebase health

**Very good.** The plugin follows a clean layered architecture (Domain / Application / Infrastructure / Presentation) with a PSR-11 IoC container and service providers. All 16 custom database tables are actively queried, all 16 PHP templates are loaded, all service provider registrations are used, and no orphaned CSS or JS dead code was found.

The six items fixed here were minor: one redundant action dispatch, one vestigial parameter, one unused enum method, two stale development comments, and one undocumented constant. None represented functional bugs or security issues.
